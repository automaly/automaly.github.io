Fix: “The Requested Performance Counter is Not a Custom Counter” in Dynamics 365 Finance & Operations
#

Environment: Dynamics 365 Finance & Operations | One-Box VHD Dev/Demo Environment
Area: General Ledger
Type: Configuration / Performance Counter Issue

The Problem
#

When attempting to create a General Ledger (GL) journal entry in a Dynamics 365 F&O one-box VHD environment, the following error appears:

The requested Performance Counter is not a custom counter, it has to be initialized as ReadOnly.

This error is commonly encountered on VHD-based development or demo environments downloaded from Microsoft Lifecycle Services (LCS). It does not typically affect production environments.

Root Cause
#

Dynamics 365 F&O uses Windows Performance Counters to monitor internal application health — such as transaction throughput, subledger activity, and tax processing metrics. These counters are registered via Instrumentation DLLs bundled with the application.

On VHD environments, these counters can become:

  • Stale — left over from a base image or previous installation
  • Corrupt — partially registered due to a failed update or snapshot
  • Mismatched — registered as system ReadOnly counters instead of custom writable ones

When D365 F&O attempts to write to a counter that Windows has flagged as ReadOnly (a system counter), the error is thrown and the GL operation is blocked.

The Fix
#

The resolution involves three steps:

  1. Stop all AOS-related services to release file locks on the DLLs
  2. Re-copy and re-register the instrumentation DLLs to the AOS bin directory, then call each module’s Setup() method to properly initialize the Performance Counters
  3. Rebuild Windows Performance Counters and restart IIS

Step-by-Step Solution
#

Prerequisites
#

  • Remote Desktop access to the VHD
  • PowerShell running as Administrator
  • D365 F&O installed at K:\AOSService\ (adjust the drive letter to match your environment)

Step 1: Stop Services
#

Before copying any DLLs, the AOS and IIS services must be stopped to avoid file lock errors.

iisreset /stop
Stop-Service -Name "DynamicsAXBatchManagement" -Force -ErrorAction SilentlyContinue
Start-Sleep -Seconds 5

Step 2: Run the Performance Counter Fix Script
#

The following script copies the correct instrumentation DLLs into the AOS bin directory and explicitly calls each module’s Setup() method to re-register the Performance Counters:

$AOSDirectory = 'K:\AOSService\PackagesLocalDirectory'
$AOSBinDirectory = $AOSDirectory + "\bin"

# Load prerequisite DLL
[Reflection.Assembly]::LoadFrom("$AOSBinDirectory\Microsoft.Diagnostics.Tracing.EventSource.dll")

# Define DLL names
$sharedDLL    = 'Microsoft.Dynamics.AX.Xpp.AxShared.dll'
$subledgerDLL = 'Microsoft.Dynamics.Subledger.Instrumentation.dll'
$taxDLL       = 'Microsoft.Dynamics.Tax.Instrumentation.dll'
$prodCfgDLL   = 'Microsoft.Dynamics.ProductConfiguration.Instrumentation.dll'
$sourceDocDLL = 'Microsoft.Dynamics.SourceDocumentation.Instrumentation.dll'

# Subledger
Copy-Item $(Join-Path $AOSDirectory -ChildPath "Subledger\bin" | Join-Path -ChildPath $subledgerDLL) -Destination $AOSBinDirectory
[Reflection.Assembly]::LoadFrom($(Join-Path $AOSBinDirectory -ChildPath $subledgerDLL))
[Microsoft.Dynamics.Subledger.Instrumentation.PerformanceCounterCatalog]::Setup()

# Tax
Copy-Item $(Join-Path $AOSDirectory -ChildPath "Tax\bin" | Join-Path -ChildPath $taxDLL) -Destination $AOSBinDirectory
[Reflection.Assembly]::LoadFrom($(Join-Path $AOSBinDirectory -ChildPath $taxDLL))
[Microsoft.Dynamics.Tax.Instrumentation.PerformanceCounterCatalog]::Setup()

# Source Documentation
Copy-Item $(Join-Path $AOSDirectory -ChildPath "SourceDocumentation\bin" | Join-Path -ChildPath $sourceDocDLL) -Destination $AOSBinDirectory
[Reflection.Assembly]::LoadFrom($(Join-Path $AOSBinDirectory -ChildPath $sourceDocDLL))
[Microsoft.Dynamics.SourceDocumentation.Instrumentation.PerformanceCounterCatalog]::Setup()

# Product Configuration
Copy-Item $(Join-Path $AOSDirectory -ChildPath "ApplicationSuite\bin" | Join-Path -ChildPath $prodCfgDLL) -Destination $AOSBinDirectory
[Reflection.Assembly]::LoadFrom($(Join-Path $AOSBinDirectory -ChildPath $prodCfgDLL))
[Microsoft.Dynamics.ProductConfiguration.Instrumentation.PerformanceCounterCatalog]::Setup()

# Shared DLL
[Reflection.Assembly]::LoadFrom($(Join-Path $AOSBinDirectory -ChildPath $sharedDLL))
[Microsoft.Dynamics.Ax.Xpp.AxShared.AxPerformanceCounters]::InitializePerformanceCounterCategories()

Step 3: Rebuild Windows Performance Counters and Restart IIS
#

lodctr /R
iisreset /start
Start-Service -Name "DynamicsAXBatchManagement" -ErrorAction SilentlyContinue

lodctr /R rebuilds the entire Windows Performance Counter registry from its backup files, clearing any stale, duplicate, or corrupted entries.

What Each Part of the Script Does
#

SectionWhat It Does
LoadFrom(EventSource.dll)Loads the tracing prerequisite needed by the instrumentation DLLs
Copy-Item ... -Destination $AOSBinDirectoryCopies the latest DLL version from its source package folder into the AOS bin
[Reflection.Assembly]::LoadFrom(...)Loads the DLL into memory so its types become available
PerformanceCounterCatalog::Setup()Re-registers the module’s Performance Counters correctly as writable
AxPerformanceCounters::InitializePerformanceCounterCategories()Initializes the core AX shared performance counter categories
lodctr /RRebuilds all Windows Performance Counters from scratch
iisresetRestarts the web server so D365 picks up the new counter registrations

Common Errors and Fixes
#

“The process cannot access the file because it is being used by another process”
#

The DLL is locked by a running service. Make sure to stop IIS and the Batch Management service before running the script:

iisreset /stop
Stop-Service -Name "DynamicsAXBatchManagement" -Force -ErrorAction SilentlyContinue
Start-Sleep -Seconds 5

If the lock persists, do a full reboot of the VHD first.

“Cannot find path … because it does not exist”
#

The DLL is not in the expected subfolder. Use this command to locate it dynamically:

Get-ChildItem -Path 'K:\AOSService\PackagesLocalDirectory' -Recurse -Filter 'Microsoft.Dynamics.ProductConfiguration.Instrumentation.dll' | Select-Object FullName

Update the Copy-Item path in the script to match the actual location on your VHD.

Notes
#

  • Drive letter may vary — Some VHDs use C:\, J:\, or K:\. Always verify your drive letter before running the script.
  • This issue is specific to VHD/one-box environments — It is caused by the nature of how VHD images are packaged and does not affect cloud-hosted or production environments.
  • Safe to re-run — The script is idempotent. Running it multiple times will not cause harm.
  • After the fix — Navigate back to General Ledger and attempt to create the journal again. The error should be resolved.
